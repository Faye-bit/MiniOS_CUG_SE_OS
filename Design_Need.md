# 简单对象存储服务



## 需求概述

设计并实现一个名为 MiniOS（Mini Object Storage）的简单对象存储服务。系统摒弃传统文件系统的层级目录结构，采用扁平化命名空间管理数据，用户通过跨进程API上传、下载和删除对象。每个对象由数据本身、元数据和全局唯一标识符三部分组成。服务器需支持多客户端并发访问，保证数据的一致性和可靠性，所有对象数据最终持久化到一个自行设计的单一复合文档文件（store.odb）中。



## 详细设计

### 对象数据存储设计

实现单个文件存储所有对象数据，内部包括：

- 超级块：位于文件头部，记录对象总数、元数据区偏移量、数据区偏移量、块大小、自由块位图起始位置等全局信息。
- 对象元数据区：存储所有对象的元数据条目，每个条目包含对象全局唯一标识符（UUID）、对象名称、大小、类型、创建时间、自定义标签、数据块指针链表等。
- 数据块区：实际存储对象内容的区域，采用固定大小（如 4KB）的块进行管理。大对象的数据被拆分成多个块，通过元数据中的块指针链表串联。
- 自由块位图：标记每个数据块的占用/空闲状态，支持块的动态分配与回收。

对象数据存储在接口上需要至少支持Put、Get、Delete、List 等方法：

- Put 函数接收上传数据流，分配足够数量的数据块，写入数据，生成 UUID，创建元数据条目，更新超级块和自由块位图；
- Get 函数根据 UUID 或对象名称查找元数据，解析块指针链表，依次从各数据块读取内容并拼接返回；
- Delete 函数根据标识符查找元数据，遍历其所有数据块，在自由块位图中标记为释放，删除元数据条目，更新超级块；
- List 函数遍历元数据区，返回所有对象的基本信息摘要。


###  类主存的共享缓冲区管理——共享内存的分页分配机制

将共享内存的数据区按固定大小的页（Page，如 4KB）进行划分，并在数据区头部维护一个页分配位图，每一位标记对应的页是空闲还是已占用。实现 alloc_page() 函数，在位图中寻找连续 N 个空闲页分配给一个对象使用，返回起始页号。实现 free_page() 函数，当对象传输完成后释放其占用的所有页，将相应位图位置零。

大对象的分页传输：当一个对象的大小超过单页容量时，客户端需要申请多页连用。对象数据被切分为若干个页大小的数据块，依次写入各页。控制区中记录该对象的页号链表（类似于文件系统中索引节点的块指针链表）。服务端读取时，根据页号链表依次从各页中读取数据并拼接为完整对象，然后写入数据文件进行持久化。当共享内存数据区无足够连续空闲页时，新请求必须等待。

### 对象读写缓存设计

服务进程中提供读写缓存，保持经常使用的数据在内存中，实现 LRU(最近最少使用)缓存策略；缓存容量可配置(如通过命令行参数)；统计并显示缓存命中率；支持缓存预热功能。

### 服务进程设计

以独立守护进程方式运行；实现进程间通信机制；支持多客户端并发访问；提供基本的启动、停止、状态查询接口。

### 客户端设计

客户端可以基于命令行也可以采用图形用户界面，能够查询展示服务状态、控制服务的启停、模拟对象存取操作、枚举对象等。



## 拓展需求（选择一两点实现即可）

### 高级实现策略

多生产者-多消费者模型；对象存储服务内部采用多线程。

### 监控与管理扩展

实现Prometheus 监控接口；添加日志系统；

### 设计聚焦，突出1-2 个点

选择 1-2 个点重点设计，但一定要保证功能完善，非重点部分可以用现有技术替代或简化。
